Neotilde — Context detection: promoted slot visuals

The keybar now has a locked left section + a horizontally scrollable region. When detection identifies a context (vim, python, less, …), context-relevant promoted keys slide into the front of the scroll region. Defaults push right but remain reachable. Promoted keys need a visual identity that reads as "this is here because of context" — distinct from default convenience slots, without overloading the press/focus/modifier-armed states.

Treatment comparison — same key, four candidate visuals

Bronze-tinted background, soft halo, top-edge accent, or combo. Glyph contrast stays high in all four — the difference is in the chrome around the glyph.

Applied — shell vs vim vs python

Same keybar across three contexts. Locked region (window pill · pane pill · arrows · Esc) never moves. Ctrl/Alt/Shift and Tab anchor the start of the scroll region. Promoted keys (bronze-tint + top-edge) appear directly after Tab. Defaults follow.

Notes

  • Glyph readability: bronze fill is low-opacity (~12%) so the glyph stays at full contrast against the panel background.
  • No state collision: halo is reserved for cursor halo / focus, brighter fill is reserved for pressed, full-bronze fill is reserved for modifiers. Promoted = tint + top accent only.
  • Engage animation: on context entry (after dwell), promoted slots slide in from left + brief one-cycle brighten of the top accent so the eye notices the change.
  • Disengage: asymmetric 1500ms — promoted slots slide out, defaults reflow left. Protects against brief :! excursions from vim.
