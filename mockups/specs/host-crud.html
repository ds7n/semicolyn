Neotilde — Host create / edit form

The CRUD form reached two ways: (1) picker row → swipe → Edit (quick), (2) Settings → Hosts → tap host (deep). Same form, same layout — quick path simply opens with everything collapsed. Schema and naming come from host-config-model-design.md. Colors and components from design-system.html.

3 — Validation surfaces

Validation is inline, never modal. Surfaced at the row level for field-scoped issues and at the section level (top of section) for cross-field issues. Save remains tappable except when a required field is empty.

Default-expansion rules
  • New host: Basics + Auth open; everything else collapsed.
  • Edit host: a section opens iff it has at least one field set to a non-default value (or, for booleans, set to a non-fallback). Otherwise collapsed.
  • Validation triggers: tapping Save runs validation and auto-expands any section containing an error or warning.
  • State pad badges: mosh "enabled" / tailscale "required" badges appear on section headers regardless of expansion — visible at a glance.

Notes on form scope

Out of scope for this mockup (separate brainstorms):
  • Delete-confirm modal — referenced jumphost case carries its own message
  • Multi-connection switching (live vs recent transitions on save)
  • Import wizard UI for ~/.ssh/config beyond the success-banner shown above
  • The Defaults editor surface — same schema minus required fields, identical visual treatment
Quick-edit vs deep-edit:
  • Same form. Quick (from picker swipe) presents as a half-sheet from the bottom; deep (from Settings) presents as a full-screen push from the right. Identical contents and layout. Reduces surface to maintain.
  • Quick-edit opens with the same default-expansion rules — typically the user lands on Basics + Auth, which is what they came for.
