Neotilde — Connection-status banner with multiple live connections

The locked banner rule is "transient, top of screen, only visible when something is wrong" — designed implicitly for one foreground session. With multiple live connections, what does the banner reflect when a background one degrades? Three options shown against the same scenario.

FG prod-web-3 — healthy (mosh, hooked into nothing weird).
BG staging-db — mosh roaming, last frame 12s ago (amber-degraded).
The user is reading tail -f app.log on prod-web-3. Question: does the user see anything about staging-db's trouble before they switch to it?

A · Foreground-pane only

Banner reflects only the active session. Background trouble is silent until you switch. Picker rows render statelessly — all "Live" looks the same.

What the user sees: no banner, no warning, no hint. The background trouble is invisible — both the terminal view and the picker show "all healthy."
Pros: minimal chrome, never wrong about the FG session, picker stays simple.
Cons: a long-degraded BG mosh that's about to give up its server-side state has no surface. User finds out only when they switch — and gets the "huh, when did this happen?" surprise.

Banner reflects the worst state across all live connections. Even while FG is healthy, BG trouble surfaces as an amber banner naming the offending host. Tap = switch.

What the user sees: an amber banner labelling "staging-db" intrudes while they're reading prod-web-3's log. Picker row also shows amber dot + "degraded" sub-text.
Pros: no surprises — every problem is surfaced as soon as it happens, with the host name explicit.
Cons: noisy. A flaky BG mosh raises a banner over an unrelated FG session. The banner now has to carry host identity, eating space. Violates the spirit of the locked rule ("only when something is wrong" with the active session).

C · Foreground banner + picker-row indicator [recommended]

Banner stays single-subject (the FG session). Background trouble surfaces as a small colored dot on the picker row — no banner intrusion until the user opens the picker or switches to the affected connection.

What the user sees: nothing while they're reading prod-web-3 — the FG session is healthy. The moment they open the picker (which they do whenever they want to switch), the amber dot + "degraded · 12s" on the staging-db row tells them what's up. If they switch to staging-db, then the amber banner slides in for that session (standard locked behavior).
Pros: respects the locked "banner = active session" rule. No FG noise from BG problems. Picker is the natural place to see "what's happening across everything." Symmetric with the existing activity-badge pattern on window switching.
Cons: catastrophic BG events (mosh-server gave up entirely → row demoted to Recent) aren't visible until the user opens the picker. Acceptable if "open picker to switch" is already common; could be paired with a one-time toast on transition to Recent (deferred).

How to read this

All three options have the same banner mechanic when the foreground session is in trouble (amber/red banner at top, identical to the single-connection design). The only question is how multi-connection awareness leaks into the FG view. C keeps the FG quiet and uses the picker — which the user opens to switch anyway — as the multi-connection awareness surface.
