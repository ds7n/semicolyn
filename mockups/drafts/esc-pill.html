Semicolyn — Fused "Esc" pill (Esc + window navigation + unified picker)

Replaces the previous separate Win pill and Esc slot with one fused pill at the leftmost locked position. The window-pill role (next/prev, picker) folds into this pill's gestures. The Esc keystroke remains the tap action.

Gesture vocabulary

One pill, five distinct gestures. The pill content is the literal text Esc with a small dim hint glyph (≡) indicating "this slot has more than tap." Window title is not displayed on the pill itself — the user knows what window they're in from terminal content, plus the pill reveals the window list on swipe-up.

New keybar layout (locked-left only)

Locked-left becomes: [Esc pill] · [Pane pill] · [arrow-pad]. The Win pill is gone — its role lives inside the Esc pill's gestures and the unified picker. Other locked decisions for Ctrl/Tab promotion still pending in the broader brainstorm.

What this trade saves and costs

Wins:
  • Locked-left shrinks by one slot — from Esc · Win · Pane · arrows to Esc · Pane · arrows (with Ctrl/Tab decisions still open).
  • The leftmost slot does triple duty: vim Esc, window nav, picker entry. Coherent identity ("the pill that opens everything").
  • Unified picker replaces two pickers (host picker + window picker). One mental model.
  • Gesture vocabulary on the pill is rich but learnable: tap = key, horizontal = window switch, vertical = window picker / create, long-press = full picker.
  • The dim ≡ glyph mirrors existing locked-design language — no new pattern introduced.
Costs:
  • Window title is no longer always visible on the keybar. The user identifies the current window by terminal content (usually obvious) or by opening a picker.
  • Tap-to-next-window is lost (old win pill behavior). Replaced by swipe-right. Muscle memory cost for current power users — but Semicolyn hasn't shipped, so the cost is theoretical.
  • Esc lives at the far-left, the least-thumb-friendly position on iPhone. Vim users press Esc a lot. The compensation is that the swipe-right / swipe-up / long-press gestures all live in the same thumb-unfriendly corner — there's no good location that's friendly for all of these.
  • Swipe-down for create-window is a less-discoverable affordance. Mitigated by the confirmation sheet teaching the gesture's meaning on first use.
