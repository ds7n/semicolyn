What we own
Predictor strip + keybar pill row. ~30% of keyboard real estate.
Pros
  • No keyboard-extension entitlement — ships under regular App Store rules
  • iOS handles alphabet, autocaps, dictation, swipe-type — for free
  • Smaller surface, simpler to get right; what every competitor ships
  • Predictor strip + pills are where 100% of the unique gestures already live
Cons
  • Symbols missing on iOS (` ~ | \ { } esc-codes) need long-press detours or a "symbols" pill
  • No control over key sizing, sticky-modifier visuals, or repeat behavior
  • Function keys (F1–F12) only via long-press or numeric overlay — never first-class

Full custom system keyboard

Glymr designs the entire keyboard region. No iOS keyboard inside Glymr — one unified terminal-optimized surface.

What we own
Every key, every modifier, every long-press. 100% of keyboard real estate.
Pros
  • Function keys, esc, ctrl, brackets, pipe — all first-class, no long-press detours
  • Sticky-modifier UX, repeat rates, key sizing — all under our control
  • Cohesive: predictor + keybar + key grid feel like one designed surface
  • The differentiator is the whole keyboard, not just a strip above it
Cons
  • ~3× the design + implementation surface; layouts × locales × iPad × landscape
  • Re-implement: autocaps, dictation handoff, swipe-type, emoji, language switching
  • Risk that users miss something familiar (dictation mic, emoji picker) and bounce
  • Slower path to v1; harder to iterate on once shipped (muscle memory)
Option C (recommended)

Hybrid · A for v1, B as v2 question

Ship the accessory bar first — the unique gestures already live there. Treat the full keyboard as a v2 decision once real usage shows where iOS fights users.

v1 (ships)
Accessory bar above iOS keyboard. Predictor + pills + small key row.
→
v2 (gated)
Full custom keyboard, if usage shows iOS friction. Pulled forward only with evidence.
Why this
  • Fastest path to validating the unique stuff (pills, predictor, gestures)
  • Defers the most expensive bet (custom keyboard) until we know it's needed
  • Predictor + pills + symbols-pill cover ~90% of pain in Option A — Option B's upside is marginal until data says otherwise
  • If v2 happens, the keybar pills and predictor strip transplant directly onto a custom keyboard region
What it commits us to
  • Living with iOS's missing symbols via a long-press / "symbols" pill for v1
  • No first-class function keys until v2
  • Need a concrete "v2 trigger" — what usage signal would actually pull the trigger?
All three phones show the same app state — terminal session with predictor surfacing claude as the top suggestion. Only the keyboard region differs.
