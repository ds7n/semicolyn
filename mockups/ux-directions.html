Brand palette — "Bell bronze" (rough)

Leveraging the name: Glymr = Old Norse "ringing sound" (a struck bell → bronze) and the English ear hears "glimmer" (faint light catching in darkness). Both meanings point at the same material: lit bronze on cool dark, with verdigris patina as a tension color. Avoids salmon-clay (Anthropic), mono-green (ChatGPT), gradient-purple ("AI"), matrix-green and phosphor-amber (terminal stereotypes), and ice-blue (Norse stereotype).

Question 1 — Window switching (locked direction)

Locked: a single pill in the keybar is the entire nav surface. Title only (truncates past ~7 chars). Four gestures: tap = next, swipe right = next, swipe left = previous, long-press = picker. Cycling wraps with a light haptic tick on wrap. A terminal-area horizontal swipe is a power-user shortcut to the same actions.

Single-finger horizontal swipe anywhere in the terminal area also switches windows — Safari-tabs-style alternative to the pill. Auto-suspends when the focused pane has mouse mode active; can be disabled globally in settings.

Question 2 — Panes (locked direction)

Locked: tmux engine; panes render as real splits; a second pill in the keybar handles all pane management so the terminal area stays available for terminal-native interactions — most importantly, iOS-native text selection (long-press → magnifier → copy).
Split terminology: horizontal split = horizontal divider, original on top, new on bottom. Vertical split = vertical divider, original on left, new on right (vim convention).

Question 3 — Cursor placement (locked direction)

Locked: a 60pt halo around the cursor is the only place that starts a cursor-drag. Drag is delta-based (mouse-like, never joystick) with mild acceleration on fast flicks. A loupe magnifies above the cursor so the user watches the cursor, not their finger. Vertical motion under 1.5 cells is clamped to horizontal — protects against the shell-history footgun (a stray vertical pixel sending an Up-arrow that swaps the current readline buffer).
Cursor and finger may decouple during accelerated travel; that's the intended mouse metaphor. Lift commits. Tap inside the halo without movement is a no-op.

Resting state

Faint halo, always visible

Halo at ~15% opacity announces the gesture. Cursor block in vim normal mode, beam in shell.

Gain curve — mouse, not joystick

Velocity-based gain

Slow drags are near-1:1 for precision. Fast flicks accelerate up to ~3× so long jumps don't require crossing the whole screen. 

  gain
   3× ┤             ╭─────────
      │           ╭─╯
   2× ┤        ╭──╯
      │     ╭──╯
   1× ┤────╯
      └─┬──────┬─────┬──────▶
        0   600pt/s  1.2k    finger speed
  • Precision in the precision zone (slow = 1:1)
  • Long jumps tractable without ergonomic flicks
  • Predictable: ratio depends on speed only, never on prior travel

Vertical dead-zone

Under 1.5 cells of cumulative vertical travel, the cursor moves on the horizontal axis only — no Up/Down arrows synthesized. Cross the threshold and vertical unlocks for the rest of the gesture. 

  shell prompt:                vim:

  $ very-long-command_         :42█
       ↑ horizontal nudge       ↑ vertical drag (2+ cells)
       (vertical drift          fully respects vertical
        suppressed; no
        history surprise)
  • Shell prompts protected from accidental history scroll
  • vim / less / TUI vertical drags still work — just need ≥1.5 cells
  • No app detection required; pure gesture heuristic

Edge cases

Cursor offscreen (scrollback)

While the user has scrolled into history, the cursor isn't on screen — nothing to grab. A small ⌖ indicator parks at the bottom edge of the active pane. Touching it engages the drag; first finger movement also auto-scrolls the viewport back to the live cursor.

Mouse mode active

In set mouse=a / htop / less, taps elsewhere in the pane pass through as mouse events. The cursor-drag halo still works — we synthesize arrow keys, not mouse events. No conflict, because the app sees keyboard input.

iOS selection in progress

Long-press selection takes precedence. While the iOS selection handles are visible, cursor-drag engagement is suppressed. Dismiss selection (tap away) to re-arm the halo.

Open subquestions: (a) Default halo opacity — 15% might be invisible on light-themed terminals; needs a theme-aware floor. (b) Haptic on lift — always, or only when ≥N cells traveled (avoid buzz on tiny adjustments)? (c) Hardware-keyboard arrow keys remain authoritative when connected; should the halo hide while a hardware kb is attached?

Banner anatomy

Severity color & behavior

Color encodes severity; same shape and position across states so it's recognizable at a glance.

Amber · degraded — reconnecting, high latency, slow frames. Terminal stays interactive.
Red · broken — disconnected, auth failed, host unreachable. Terminal dimmed.
Green · momentary "reconnected" toast (~1.5s) before fading out. Confirms recovery.

Lifecycle

Banner is event-driven, not polled. Slides in/out with a short spring (~180ms). 

  healthy ──blip──▶ amber (reconnecting…)
       ◀──recover─┤
                  └──timeout──▶ red (disconnected)
                                  │
                                  ├──user taps──▶ reconnect attempt
                                  └──recover──▶ green toast (1.5s) ──▶ healthy

Open subquestions: (a) Swipe-up to dismiss a persistent banner — does the banner re-appear on state change, or stay dismissed for the rest of the session? (b) The expanded detail view on tap — what does it actually show? (mosh frame counter, RTT graph, last-good timestamp, host name?) (c) Multiple simultaneous problems (e.g., reconnecting on host A while host B times out) — stack banners, or show the most-severe only?
