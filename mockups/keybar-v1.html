Glymr — Keybar v1 (first try)

Evidence-grounded where the data allows it, marked as a guess where it doesn't. This is a working draft — we expect to learn from real use and change it. The whole point of first-class customization (long-press any slot to edit) is that defaults don't have to be perfect to ship. Iterate, don't over-engineer.

Design principles (lock first, debate second)

  1. Keybar's purpose (narrow): (a) window/pane pills, (b) keys iOS lacks entirely (Esc, Tab, Ctrl/Alt/Shift, arrows), (c) convenience defaults for symbols that are reachable via iOS's 123 layer but worth promoting (parens, pipes, tildes, etc.). The keybar is NOT a complete symbol surface — iOS's 123 layer still exists.
  2. Core slots vs convenience slots. Core slots (purpose a/b) ship locked. Convenience slots ship enabled but the user can remove or replace any of them.
  3. Three actions per slot: tap = primary, swipe-up = secondary, swipe-down = tertiary. Long-press = edit/customize. Exception: the arrow-pad slot uses drag-in-direction instead.
  4. Tap = highest-frequency action in the slot. Swipes are the rarer members of the same logical group.
  5. Group by logic on each slot, not by frequency across slots. Brackets live with brackets, even if a bracket would be hit more often than another slot's tap.
  6. Every slot's swipe chars are visible (small, muted, top/bottom edges) — discoverability is non-negotiable.
  7. Ctrl, Alt, Shift are sticky-for-one-keystroke. Tap → armed for the next keystroke only, auto-disarms after. Esc and Tab are normal keys — fire on tap.
  8. v1 scope: in-app accessory bar above iOS's native keyboard. iOS owns the letters; the 123 layer remains the fallback for any symbol not promoted to the keybar. Custom keyboard (with long-press alts on letters) deferred as a potential v2.
  9. User customization is first-class: long-press any slot to edit. The default set ships defensible; users own the bar.
  10. Expect to change this. The defaults are a v0 best-guess. Once Glymr has real users, the predictor's keystroke telemetry (with consent) tells us what's actually pressed. Plan for v1.5 default-set revisions; don't over-optimize now.

Slot anatomy

\ / ?
Center (large):
Tap action. The most-used key in this slot's group.
Top edge (small, muted):
Swipe-up action. Second-most-used in the group.
Bottom edge (small, muted):
Swipe-down action. Third in the group, or empty.
Long-press:
Opens the slot editor — rebind any of the three, replace the slot, or pin a new macro.

The 12-slot default — rendered

All 12 slots fit the keybar with horizontal scroll. Window and pane pills stay sticky on the left. On a standard iPhone width, ~7 slots are visible without scrolling; the rest are reachable with a small horizontal scroll. Slots with empty swipes (Tab, Esc) are intentional — those keys have no natural peer in their group; users can fill the swipes via the editor.

What's "data-backed" vs "guessed" — colored honestly

Green = backed by data I found. Amber = defensible guess; the public data was silent.

Data-backed

  • Ctrl, Tab, Esc, ↑ in default set, each as their own tap — iOS lacks them entirely, shell needs them
  • |, ~, /, \ in default set — universal shell use, deeply buried on iOS
  • ( ) in tap position — Programmer Dvorak's "index finger" priority
  • { } [ ] in swipe positions of paren slots — paired-bracket logic
  • Tap = highest-frequency principle — universal layout-design wisdom

Defensible guesses

  • ← → ↓ priority — readline conventions, no measured iOS data
  • < > as their own slot — promoted from buried-swipe under brackets; relative frequency to other slots unmeasured
  • Word-jump on swipe-up of arrows — readline Alt+arrow convention
  • $ * on slot 9 vs. somewhere else — vars/globs are common, but exact rank unknown
  • 12 slots as the right default count — not 10, not 14

Things explicitly deferred

  • Function keys (F1–F12) — none of the 10 default slots covers them. Candidates: long-press on the number row (when number row exists), or a "Fn" slot the user can pin. For v1 they live in the slot editor as buildable macros.
  • Context-swapping layouts (vim shows hjkl prominent, tmux shows prefix) — deferred to its own conversation. v1 keybar is static.
  • Per-host pinning — same as snippets, eventually a host can have its own pinned subset. v1 = one global keybar.
  • iPad layout — keybar pill sizes and positions will change with iPad real estate. v1 = iPhone-first.
  • Symbol-pill overflow drawer — what happens when user pins 30 slots? Long-scroll vs. category picker. v1 = horizontal scroll only.
